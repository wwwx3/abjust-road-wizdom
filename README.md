# Abjust — Bangkok Traffic Risk Triage

**Abjust** คือโปรเจกต์ civic-tech สำหรับ Bangkok Hackathon 2026 ที่ออกแบบมาเป็น **Traffic-Risk Intelligence Layer** เพื่อช่วยเปลี่ยนรายงานปัญหาจราจรของประชาชนให้กลายเป็นเคสที่จัดลำดับความสำคัญ มีเจ้าภาพ ติดตามได้ และนำไปใช้วางแผนเมืองต่อได้

> Abjust ไม่ได้มีเป้าหมายเพื่อแทนที่ระบบเดิมของเมือง แต่เป็นชั้นเสริมที่ช่วยให้ข้อมูลจากระบบรายงานเดิมถูกนำไปใช้ต่อได้อย่างมีประสิทธิภาพมากขึ้น

---

## แนวคิดหลักของโปรเจกต์

ปัญหาของเมืองไม่ได้อยู่แค่ “ประชาชนแจ้งเรื่องไม่ได้” แต่อยู่ที่หลังจากแจ้งแล้ว:

* รายงานจำนวนมากอาจซ้ำกัน
* แยกไม่ออกว่าเคสไหนเร่งด่วนที่สุด
* เคสอาจไม่มีเจ้าภาพที่ชัดเจน
* หน่วยงานอาจส่งต่อกันไปมา
* เคสที่ค้างอาจหายไปจากการติดตาม
* ผู้บริหารเมืองอาจไม่เห็นภาพรวมเชิงพื้นที่ของปัญหาซ้ำ

Abjust จึงออกแบบมาเพื่อช่วยให้ทุกเคสมีเส้นทางที่ชัดเจน ตั้งแต่การรับรายงานจนถึงการติดตามผลและการวางแผนเมือง

---

## ฟีเจอร์หลัก

### 1. Citizen Report

ประชาชนสามารถแจ้งปัญหาจราจร เช่น

* รถจอดกีดขวาง
* รถจอดขวางทางฉุกเฉิน
* สัญญาณไฟจราจรผิดปกติ
* ทางเดินเท้าถูกกีดขวาง
* น้ำท่วมถนนกระทบการจราจร
* ป้ายหรือเส้นจราจรไม่ชัดเจน

พร้อมระบุตำแหน่ง รายละเอียด และหลักฐานจำลอง

---

### 2. AI-assisted Summary

ระบบช่วยสรุปเหตุการณ์ให้อยู่ในรูปแบบที่เจ้าหน้าที่อ่านง่ายขึ้น

ใน prototype นี้ใช้ **rule-based AI fallback** และ mock data เพื่อการสาธิตเท่านั้น ยังไม่ใช่ AI production model

---

### 3. Risk Score และ Priority Score

ระบบประเมินความเสี่ยงและลำดับความสำคัญของเคสแบบตรวจสอบย้อนกลับได้

* **Risk Score** ใช้วัดว่าเหตุการณ์อันตรายแค่ไหน
* **Priority Score** ใช้ช่วยจัดคิวว่าเคสไหนควรดำเนินการก่อน

แนวทางนี้ถูกออกแบบให้โปร่งใส ไม่ใช่ black-box model เพื่อให้เจ้าหน้าที่และผู้บริหารเข้าใจเหตุผลของการจัดลำดับ

---

### 4. Duplicate Grouping

ระบบรวมรายงานที่ซ้ำกันหรือใกล้เคียงกันเข้าเป็นเคสเดียว เพื่อลดงานซ้ำของเจ้าหน้าที่

ใน UI ใช้คำว่า:

**จำนวนรายงานที่ถูกรวม**

ไม่ใช่ “จำนวนผู้ได้รับผลกระทบ” เพราะ prototype นี้ยังไม่ได้วัดจำนวนคนที่ได้รับผลกระทบจริง

---

### 5. Officer Dashboard

เจ้าหน้าที่สามารถดูเคสในรูปแบบ dashboard / Kanban board โดยจัดเรียงตามสถานะและความเสี่ยง เช่น

* รับเรื่องแล้ว
* กำลังตรวจสอบ
* มอบหมายหน่วยงานแล้ว
* กำลังดำเนินการ
* แก้ไขเสร็จสิ้น

---

### 6. Case Ownership Workflow

ทุกเคสควรมีเจ้าภาพที่ชัดเจน เช่น

* หน่วยงานหลัก
* หน่วยงานร่วม
* ผู้รับผิดชอบปัจจุบัน
* ขั้นตอนถัดไป
* เวลากำกับการดำเนินงาน

เป้าหมายคือป้องกันไม่ให้เคสถูกส่งต่อไปมาโดยไม่มีผู้รับงานต่อ

---

### 7. Escalation Ladder & Audit Trail

Abjust ไม่ได้หยุดแค่การแจ้งเตือน เพราะการแจ้งเตือนอาจถูกละเลยได้

ระบบจึงออกแบบให้เคสที่ค้างเกินกำหนดถูกส่งต่อเป็นลำดับขั้น เช่น

1. หน่วยงานเจ้าของเคส
2. ผู้ประสานงานกลาง
3. หัวหน้าหน่วยงาน
4. Dashboard ผู้บริหารเมือง
5. ภาพรวมสาธารณะในระดับ aggregate

ทุกการเปลี่ยนสถานะถูกบันทึกใน Audit Trail เพื่อให้เห็นว่าเคสติดอยู่ที่ขั้นตอนไหนและต้องการการตัดสินใจจากระดับใด

ระบบนี้ไม่ได้ใช้แต้มลงโทษเจ้าหน้าที่ ไม่จัดอันดับเจ้าหน้าที่ และไม่เน้นการกล่าวโทษ แต่ใช้ความโปร่งใสของกระบวนการเพื่อป้องกันไม่ให้เคสหายไปจากระบบ

---

### 8. Citizen Timeline

ประชาชนสามารถติดตามความคืบหน้าของเคสได้ เช่น

* รับเรื่องแล้ว
* ระบบสรุปเหตุการณ์แล้ว
* มอบหมายหน่วยงานที่เกี่ยวข้อง
* อยู่ระหว่างการประสานงาน
* กำลังดำเนินการ
* แก้ไขเสร็จสิ้น

Citizen view จะแสดงเฉพาะข้อมูลที่เหมาะสม ไม่แสดงรายละเอียด internal conflict ระหว่างหน่วยงาน

---

### 9. Dashboard ผู้บริหารเมือง

ผู้บริหารเมืองสามารถดูภาพรวมของปัญหาจราจรในระดับพื้นที่ เช่น

* พื้นที่ที่มีปัญหาซ้ำ
* จุดเสี่ยงสูง
* เคสค้างเกินกำหนด
* พื้นที่ที่ควรติดตามเชิงนโยบาย
* heatmap / hotspot ของกรุงเทพฯ
* ข้อมูลเชิงพื้นที่เพื่อช่วยจัดลำดับการใช้ทรัพยากร

ฟีเจอร์นี้ช่วยเปลี่ยนรายงานที่กระจัดกระจายให้กลายเป็นข้อมูลประกอบการตัดสินใจเชิงเมือง

## โครงสร้างการทำงานของระบบ

```text
Citizen Report
→ AI-assisted Summary
→ Risk / Priority Score
→ Duplicate Grouping
→ Suggested Lead Unit
→ Case Ownership
→ SLA Tracking
→ Auto-Escalation
→ Audit Trail
→ Citizen Timeline
→ Executive Map / Analytics
```

---

## การเชื่อมต่อกับระบบจริงในอนาคต

Abjust ถูกออกแบบให้เป็น integration layer ไม่ใช่ระบบที่บังคับให้ทุกหน่วยงานเปลี่ยน workflow เดิมทันที

แนวทางในอนาคต:

1. เริ่มจาก sandbox หรือ mock data
2. ทดลองกับหนึ่งเขตหรือหนึ่งประเภทปัญหา
3. เชื่อมต่อแบบ read-only กับระบบเดิมก่อน
4. พัฒนาเป็น two-way API สำหรับ status, timeline, ownership และ escalation
5. เปิดข้อมูลภาพรวมแบบ anonymized / aggregated สำหรับ public dashboard หรือ open data

## Open Source Scope

Repository นี้เปิดเผยเพื่อการเรียนรู้ การพัฒนาต่อยอด และการสาธิต civic-tech prototype

สิ่งที่สามารถเปิดเผยได้:

* source code
* UI prototype
* API schema
* risk scoring logic
* workflow logic
* mock data
* documentation
* dashboard template

สิ่งที่ไม่ควรเปิดเผย:

* ข้อมูลประชาชนจริง
* ชื่อ เบอร์โทรศัพท์ หรือข้อมูลระบุตัวบุคคล
* ภาพหลักฐานจริงที่มีใบหน้า/ป้ายทะเบียน
* API keys
* private tokens
* internal officer notes
* private government data

## Privacy & Ethics

โปรเจกต์นี้คำนึงถึงความเป็นส่วนตัวและความปลอดภัยของข้อมูล

แนวทางที่ควรใช้หากพัฒนาต่อจริง:

* ใช้ข้อมูลเท่าที่จำเป็น
* เบลอใบหน้าและป้ายทะเบียนก่อนจัดเก็บหรือแสดงผล
* จำกัดสิทธิ์การเข้าถึงข้อมูลตามบทบาท
* เปิดเผยเฉพาะข้อมูลภาพรวมต่อสาธารณะ
* บันทึก Audit Trail เพื่อความโปร่งใส
* ให้มนุษย์เป็นผู้ตัดสินใจขั้นสุดท้ายในเคสสำคัญ
* ใช้ AI เป็นเครื่องมือช่วยตัดสินใจ ไม่ใช่ผู้ตัดสินแทนเจ้าหน้าที่

---

## ข้อจำกัดของ Prototype

Prototype นี้พัฒนาขึ้นภายในระยะเวลาจำกัดสำหรับ hackathon

ข้อจำกัดปัจจุบัน:

* ใช้ mock data
* ใช้ rule-based AI fallback
* ยังไม่ได้เชื่อมต่อกับระบบจริงของเมือง
* ยังไม่มีระบบ authentication จริง
* ยังไม่มี computer vision สำหรับวิเคราะห์ภาพจริง
* ยังไม่มี privacy blur อัตโนมัติ
* ยังไม่มี LINE OA integration จริง
* ยังไม่ได้ผ่านการทดสอบกับ workflow จริงของหน่วยงานรัฐ

---

## แนวทางพัฒนาต่อ

* เชื่อมต่อกับระบบรายงานเมืองผ่าน API
* เพิ่ม authentication สำหรับประชาชนและเจ้าหน้าที่
* พัฒนา LLM สำหรับสรุปรายงาน
* ใช้ computer vision วิเคราะห์ภาพหลักฐาน
* เพิ่ม privacy blur สำหรับใบหน้าและป้ายทะเบียน
* เชื่อมต่อ LINE OA สำหรับแจ้งเตือนประชาชน
* ใช้ geospatial clustering เพื่อรวมรายงานซ้ำแม่นยำขึ้น
* พัฒนา open data dashboard แบบ aggregate
* ทดลอง pilot กับเขตหรือหน่วยงานที่เกี่ยวข้อง
